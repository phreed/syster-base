= SysML v2 Grammar Coverage Punch List
:toc:

Gaps found by comparing MontiCore's official SysMLv2 grammar
(`SysMLBasis.mc4`, `SysMLParts.mc4`, `SysMLActions.mc4`, `SysMLStates.mc4`,
`SysMLConstraints.mc4`, `SysMLExpressions.mc4`, `SysMLv2.mc4`, release/7.8.x)
against the Rust parser in `src/parser/grammar/`.

Checkboxes are unchecked; check them off as items are fixed.

== Entirely missing (no token/keyword exists at all)

* [ ] `timing` keyword (`SysMLCausality`) -- Parts. No `TIMING_KW` anywhere.
* [ ] `exists ... :` (`ExistsExpression`) -- Expressions. No `EXISTS` token.
* [ ] `union` as a binary expression operator (`UnionExpression`) -- Expressions.
  Only `UNIONS_KW` (the unrelated relationship keyword) exists.
* [ ] `INF` / `-INF` literal (`Infinity`) -- Expressions. No infinity token/literal.
* [ ] `nat` primitive type (`MCPrimitiveTypeWithNat`) -- Expressions. Not lexed.
* [ ] LaTeX-style set operators: `\subseteq`, `\subset`, `\supseteq`, `\supset`,
  `\sqsubseteq`, `\in` -- Expressions. None exist.
* [ ] `static` modifier -- Basis. Not even lexed as a token; falls through as `IDENT`.
* [ ] Symbolic visibility shorthand `+` / `-` / `#` for public/private/protected --
  Basis. Tokens exist but are never checked as visibility markers.
* [ ] `?` as a `readonly` alias -- Basis. `QUESTION` token exists (used for
  ternary) but is never treated as readonly.
* [ ] `refines` (`SysMLRefinement`) -- Basis. No dedicated keyword path found.
* [ ] `AnonymousReference` production -- Basis. Not distinguished from
  `AnonymousUsage`; no distinct implementation.
* [ ] `StateDef` / `StateUsage` parameter list, e.g. `state def S (in x: Integer) { }`
  -- States. Confirmed by test: real syntax error.
* [ ] `first X if guard then Y` (guard attached to a `first` succession) --
  Actions. `parse_first_action` has no guard branch.

== Real bugs (should parse, doesn't -- or silently mis-parses)

* [ ] `state s parallel { ... }` -- confirmed empirically: `parallel` is
  swallowed as the name of a *second, unrelated* usage instead of a modifier
  on the state. `actions.rs:613` has the check but the `usage.rs` dispatch
  path never reaches it.
* [ ] `do action1()` inside a transition -- call-style effects break parsing
  and corrupt the rest of the transition (the `then` target is dropped into
  an `ERROR` node). Only bare-name / `send` / `accept` / `assign` / inline
  `action` effects work (`states.rs:196-214`).
* [ ] `entry Off.entry;` and other dotted qualified-name state-action bodies
  -- `parse_state_subaction` uses `parse_identification` (single name)
  instead of `parse_qualified_name`. Confirmed syntax error on `.`. Same
  code path affects `DoAction` and `ExitAction`.
* [x] `assume requirement r1 {...}` / `require requirement r1 {...}` --
  mis-dispatched into `parse_requirement_constraint`, which only checks for
  `constraint`, not `requirement`; corrupts the tree. +
  _Fixed: `entry.rs`'s `ASSERT_KW`/`ASSUME_KW`/`REQUIRE_KW` dispatch now also
  checks for `verify`/`requirement` (not just `satisfy`/`not`), and
  `parse_requirement_verification` now consumes any of
  `assert`/`assume`/`require` as its leading modifier instead of only
  `assert`. As a side effect this also fixes the analogous
  `assume satisfy r1;` / `require satisfy r1;` mis-dispatch. `not` is kept as
  an unconditional trigger for the verification path (not narrowed to
  `not satisfy`/`not verify`) specifically to avoid regressing
  `assert not massLimitation {...}`-style constraint usages, which
  `parse_requirement_constraint` cannot parse (see the `assert not
  constraint c1 {...}` item below, still open). Regression tests added in
  `tests/tests_parser_requirements.rs`
  (`test_requirement_usage_prefix_dispatch`)._
* [ ] `assert not constraint c1 {...}` -- the `["not"]?` modifier is never
  checked in `parse_requirement_constraint` (`requirements.rs:119`).
* [ ] `frame Pkg::Concern1;` (`ConcernReference`) -- uses
  `parse_optional_identification` (single name) instead of a qualified-name
  parser; multi-segment names likely break.
* [ ] `#Foo::Bar` (`UserDefinedKeyword`) -- only consumes `#Foo`; the
  `::Bar` remainder dangles unconsumed.
* [ ] `expose Pkg::* [@Filter]` -- the filter-bracket clause has no parser
  support, unlike the equivalent `import` path.
* [ ] Connector/succession `Endpoint` -- only reference-subsetting
  (`::>` / `references`) is supported on endpoints; general `:>`, `:>>`, `:`
  specializations on an endpoint aren't parsed.
* [ ] `@ Type` (`KerMLHasTypeSelfExpression`, symbolic self-form) -- the
  keyword self-form (`hastype Type`) works, but bare `@ Type` is routed to
  metadata-access parsing instead.

== Partial / structural concerns (parses, but with gaps or duplication)

* [ ] Payload parsing (`name:Type` / `when` / `at` / `after`) is implemented
  three separate times (`helpers.rs`, `states.rs`, `actions.rs`) with no
  shared function -- consistency risk for future changes.
* [ ] Control nodes `fork` / `join` / `merge` / `decide` only parse an
  optional name, no typing (`: TypeName`).
* [ ] `SysMLNullLiteral` / `SysMLEmptyExpression` (`()`) parse without error
  but have no distinct AST node -- indistinguishable from a degenerate
  parenthesized expression.
* [ ] Dead `SyntaxKind` variants declared but never constructed:
  `ACTION_DEFINITION`, `CALC_DEFINITION`, `ACTION_USAGE`, `CALC_USAGE`,
  `CONSTRAINT_DEFINITION`, `REQUIREMENT_DEFINITION`, `CONSTRAINT_USAGE`,
  `REQUIREMENT_USAGE` -- everything routes through generic `DEFINITION` /
  `USAGE` nodes with boolean flags instead. Not a parse bug, but any
  downstream HIR/AST code matching on these specific kinds will never see
  them.
* [ ] `FlowPayload` isn't a first-class production -- payload params are
  parsed as generic typed parameters.
* [ ] Bare `ordered` / `nonunique` cardinality-modifier parsing is
  copy-pasted across roughly six call sites rather than shared -- fragile
  for future usage kinds that forget to copy the snippet.

== Confirmed fully implemented

Everything else across all seven grammar files checked out as fully
implemented, including:

* Definitions/usages for `part`, `attribute`, `port`, `connection`,
  `interface`, `flow`, `allocation`, `occurrence`, `item`, `analysis`,
  `verification`, `case`, `view`, `viewpoint`, `rendering`, `metadata`.
* Requirements/constraints/concern long forms and the shorthand
  `satisfy` / `verify` forms.
* The full expression precedence chain: collect (`.{ }`), select
  (`.?{ }`), arrow invocation (`->`), cast (`as`), `istype`, `hastype`
  (infix), ternary, index (`#(...)`), power (`^`, `**`), range (`..`),
  and/or/xor/not.
